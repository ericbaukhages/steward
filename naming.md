# Naming

> This document captures naming philosophy and open questions. It is intentionally incomplete.

## Goals

Names should reinforce the philosophy of the project.

Avoid names that imply:

- AI replacing the user
- corporate productivity software
- gimmicks or novelty

Instead, names should feel:

- calm
- intentional
- long-lived
- practical
- human
- trustworthy
- optimistic

The system should feel like a well-made tool.

---

# Current Direction

## Steward

Current working name.

The name represents the philosophy of the system.

A steward does not make decisions for someone else.

A steward:

- keeps records
- maintains systems
- organizes information
- preserves history
- helps without taking ownership

This aligns well with the project's goals.

The user remains in control.

---

# Naming Philosophy

The project should not be centered around hardware.

Instead:

```
Steward
    ├── Desktop
    ├── Web
    ├── Handheld
    ├── Dashboard
    └── Mainframe
```

The software is the product.

Hardware is simply one way to access it.

This avoids the need to invent product names for every form factor.

---

# Science Fiction

Science fiction is an acceptable source of inspiration.

The project is inspired by optimistic visions of personal computing.

Themes include:

- exploration
- curiosity
- capable personal technology
- calm interfaces
- tools that augment people instead of replacing them

However, names should avoid direct references to existing fictional universes.

The goal is for the project to establish its own identity rather than borrow someone else's.

## Avoid Direct References

The project originally used Star Trek terminology as development shorthand.

Examples:

- PADD
- LCARS
- Tricorder

These were useful while exploring ideas, but should not become the public identity of the project.

The intention is to capture the *feeling* of those devices, not recreate them.

Someone discovering Steward should appreciate it on its own merits without needing to understand its inspirations.

# Naming Rules

Potential guidelines:

- Prefer nouns over verbs.
- Prefer existing English words.
- Avoid invented spellings.
- Avoid obvious technology buzzwords.
- Avoid names that sound like AI assistants.
- Avoid unnecessary acronyms.
- Avoid names tied to specific hardware.
- Prefer timeless language over futuristic language.
- Prefer names with a clear meaning outside of technology.

---

# Branding Direction

The emphasis should be on **Steward**.

Hardware should be considered clients of the system rather than separate products.

Examples:

```
Steward Desktop
Steward Web
Steward Handheld
Steward Console
Steward Dashboard
```

This keeps the ecosystem cohesive while allowing hardware to evolve over time.

If individual hardware names are eventually introduced, they should reinforce the overall philosophy rather than compete with the Steward identity.

# Open Questions

## Should hardware have product names?

### Option A

Everything is simply Steward.

Examples:

- Steward Desktop
- Steward Pocket
- Steward Console

Advantages:

- Unified identity
- Less naming overhead
- Easier documentation

---

### Option B

Each hardware family has its own identity.

Example:

```
Steward
└── Fieldbook

Steward
└── Console
```

Advantages:

- More personality
- Easier to discuss hardware individually

Disadvantages:

- More branding complexity
- Risk of fragmenting the ecosystem

# Notes

A recurring realization throughout development:

The project is not trying to build a better phone.

It is trying to build a better relationship with personal computing.

The hardware exists to support that vision.

The software is the product.

Names should reinforce that philosophy while leaving room for the project to grow into its own identity.
